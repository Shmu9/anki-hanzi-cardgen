# System Design

## Purpose

This application is a web-based Chinese character dictionary explorer and flashcard authoring tool built around the generated `dictionary/dict.sqlite3` database. It must:

- expose the generated character dictionary data in a usable interface
- visualize character decomposition and radical/component breakdowns
- generate editable flashcards and deck content
- enrich cards with audio, stroke-order media, and mnemonics
- support per-user accounts and learning preferences
- integrate with local Anki via AnkiConnect
- support manual export/import workflows when live sync is unavailable

The initial dictionary scope is character-first. The generated database currently provides glyph records, Unihan-derived meanings/readings, stroke and radical-stroke metadata, HanziDB frequency/HSK metadata, and IDS decomposition data. The architecture should leave room for later word dictionary entries, example sentences, sentence audio, and richer media without requiring a rewrite.

The target stack is:

- frontend: TypeScript, React, Tailwind CSS
- backend: Java
- scripting and ingestion utilities: Python
- data storage: SQLite

## High-Level Architecture

The system should be organized into five layers:

1. **Frontend**
   - React SPA for search, browsing, flashcard editing, media review, and sync management.
   - Tailwind CSS for styling and design system primitives.

2. **Backend API**
   - Java service exposing REST endpoints for dictionary search, entry retrieval, flashcard generation, account management, user preference management, deck management, media ingestion state, and Anki integration.
   - Owns business rules, parsing, normalization, caching, provenance, prompt construction, and conflict handling.

3. **Data Layer**
   - Read-only runtime access to generated `dictionary/dict.sqlite3` for character dictionary content.
   - Separate app SQLite database for drafts, settings, cache metadata, sync state, and sourced assets.

4. **Ingestion and Enrichment Workers**
   - Python scripts for scraping, transformation, asset normalization, batch imports, and any source-specific logic that is faster to iterate outside the Java service.
   - Can be invoked by the backend as background jobs.

5. **External Integrations**
   - AnkiConnect for local desktop Anki operations.
   - External websites for sourced audio, stroke order, and mnemonic material.
   - TTS and LLM providers as fallbacks only.
   - Locally hosted Gemma 4 fine-tuned model for mnemonic generation.

## Core Data Sources

### 1. Generated Character Dictionary Database

The current `dictionary/dict.sqlite3` schema contains:

- `metadata`: build/source metadata
- `glyphs`: canonical glyph records, including Unicode/token identity, definitions/readings, frequency/HSK metadata, stroke metadata, radical-stroke metadata, and immediate IDS decomposition
- `character_summary`: app-friendly character rows for Unicode glyphs
- `glyph_component_tree`: recursive decomposition tree for every component depth
- `glyph_base_components`: recursive base/leaf component rows with debug traversal details
- `glyph_base_components_with_definition`: base components biased toward meaningful Unicode glyphs
- `glyph_base_component_summary`: compact base component lookup for application queries

Important observations:

- `dict.sqlite3` is generated by `dictionary/scripts/build_character_db.py` from local source files.
- dictionary content is character-first, not word-first.
- `glyphs.decomp_type` and `glyphs.decomp_components` store immediate IDS decomposition details.
- recursive component views should be used for decomposition, radical/component breakdowns, and mnemonic prompt grounding.
- search should use indexed fields on `glyphs` and `character_summary` instead of naive table scans whenever possible.
- future word dictionary, example sentence, audio, and media sources should be modeled as additional sources rather than mixed directly into the generated character database.

### 2. Application Database

Use a second SQLite database for application-owned state. Suggested tables:

- `users`
- `user_sessions`
- `search_cache`
- `entry_cache`
- `flashcard_drafts`
- `flashcard_templates`
- `decks`
- `deck_sync_records`
- `media_assets`
- `media_asset_variants`
- `media_sources`
- `character_enrichments`

This keeps generated `dict.sqlite3` reproducible and separable from user-generated state.

## Functional Modules

## 1. Dictionary Explorer

Capabilities:

- search by Hanzi, pinyin, glyph token, Unicode value, internal id, and normalized text
- browse entry details
- inspect parsed and raw source data
- navigate related characters and reusable components
- filter by HSK, stroke count, radical-stroke data, frequency, and extracted metadata
- inspect immediate and recursive character decomposition
- view radical/component breakdowns with available meanings and metadata

The frontend should use a split-pane desktop layout:

- left: search and result list
- right: entry detail panel

On mobile:

- search page
- result list
- entry detail page
- decomposition detail page or tab

## 1a. Accounts and Personalization

Capabilities:

- create and sign in to a user account
- store per-user learning preferences
- persist radical meaning preferences per character component
- maintain multiple preferred meanings per radical
- choose a default mnemonic generation profile

This is required because mnemonic quality depends heavily on the learner's preferred internal vocabulary for components.

## 2. Flashcard Builder

Capabilities:

- add entries to a staging area
- choose per-entry card template
- generate card fields from parsed dictionary data
- attach audio, stroke order, and mnemonic enrichments
- edit fields before export or sync
- preview resulting note layout
- generate mnemonics using the signed-in user's radical preferences

Templates should include:

- character recognition
- character production
- single-character study
- decomposition/radical breakdown study
- future vocab recognition
- future vocab production
- custom template mapping

## 3. Media and Enrichment Pipeline

This is a first-class subsystem, not an afterthought.

### Media Types

- character-level audio
- decomposition or radical breakdown HTML/media for character cards
- stroke-order diagrams or animations
- mnemonic text
- optional word-level audio later
- optional example sentence audio later

### Source Priority

For each asset type, the system should use a ranked strategy:

1. trusted website sources with explicit provenance
2. previously cached validated assets
3. user-supplied assets
4. generated fallback assets

### Audio Strategy

For each character and term:

1. attempt to resolve audio from a source website
2. if unavailable, check prior cache
3. if still unavailable, generate TTS
4. mark generated assets as fallback and lower-confidence

### Stroke Order Strategy

For each character:

1. fetch stroke order media from an approved source website
2. normalize to supported formats such as SVG, GIF, or MP4
3. cache variants for web preview and Anki export
4. if unavailable, record no-source state rather than fabricating strokes

Stroke order should not be LLM-generated.

### Mnemonic Strategy

For each character:

1. attempt to ingest mnemonic text from an approved source website
2. allow manual author override
3. if unavailable, generate mnemonic candidates with a locally hosted fine-tuned Gemma 4 LLM
4. mark generated mnemonics as synthetic and reviewable

Mnemonic quality improves if generation is grounded with:

- parsed dictionary meaning
- pronunciation
- radical and stroke metadata
- component segmentation where available
- IDS decomposition tree and base component summary
- user-selected radical meanings
- user-selected alternate radical meanings when the generation profile allows them

### Radical Preference-Aware Mnemonic Generation

Mnemonic generation should be personalized and account-scoped.

For a character such as `郎`, the generation request should be built from:

- the character meaning and usage
- pronunciation and optional sound hints
- extracted component structure such as `良` and right-side `阝`
- the current user's preferred meanings for those components
- optionally additional user-approved alternates for the same component

Example prompt payload shape:

- target character: `郎`
- target glosses: `young man`, `gentleman`, `official`, `husband in compounds`
- sound hint: `lang2`
- components:
  - `良` = `good, fine`
  - `阝 (right side)` = user default `town / district`
  - alternates enabled for this user: `hill`, `slope`, `ear`
- selected mnemonic structure template: `scene-based story`

The model should then generate mnemonic candidates that explicitly use the learner's preferred language rather than a generic canonical radical gloss.

### Mnemonic Structure Templates

The application should provide a small curated set of mnemonic structures that users can pick from before generation.

Examples:

- `Scene Story`
  - combine components into a single vivid mental scene
- `Component Ladder`
  - explain each component in sequence and tie them to the target meaning
- `Sound Hook`
  - use pronunciation as a major memory anchor
- `Compact Cue`
  - produce a short, low-word-count mnemonic for quick review
- `Humorous Image`
  - bias toward absurd or exaggerated imagery for memorability

The selected structure template should become part of the prompt input and part of the cache key.

### Local LLM Service

The mnemonic generation system should call a locally hosted model service owned by this application stack.

Recommended shape:

- Java backend owns the mnemonic generation API and prompt assembly
- local model runtime hosts a fine-tuned Gemma 4 model
- Python scripts can handle prompt experimentation, evaluation, and batch generation utilities

The Java backend should not let the frontend prompt the model directly. The backend must:

- validate the user's selected radical meanings
- validate the selected mnemonic structure template
- build a structured prompt
- store the exact prompt inputs and model version
- record which user preference set was used
- cache generated candidates by character, user preference hash, and model version

### Rate Limiting and Retries

Mnemonic generation should be protected by account-level and system-level rate limits.

Recommended controls:

- per-user request rate limit
- per-user concurrent generation limit
- global model queue limit
- optional per-IP rate limit for unauthenticated or abuse-prone traffic

The backend should return clear rate-limit state such as:

- remaining quota
- reset time
- current queue position when relevant

Retry behavior should be explicit:

- manual retry from the UI
- automatic retry for transient local model failures
- capped retry count
- idempotent regeneration requests when the inputs are unchanged

Retries should reuse the same prompt inputs unless the user changes:

- radical preferences
- mnemonic structure template
- pronunciation hint settings
- target gloss selection

### Provenance and Review

### Provenance and Review

Every enrichment record must preserve:

- source URL
- fetch timestamp
- parser version
- license status
- fallback indicator
- review status
- user account when user-specific generation was involved
- model name and model version for generated mnemonics
- radical preference snapshot or hash used for the generation run
- mnemonic structure template id
- retry count and final generation status

This prevents generated and sourced content from being conflated.

## 4. Deck Management

Capabilities:

- create and edit local deck mappings
- define note models and field mappings
- review draft notes before sync
- detect duplicates and conflicts
- update synced notes safely

Deck logic should treat local drafts as the source of truth and Anki as a synchronized target.

## 5. Anki Integration

Primary integration is local desktop Anki through AnkiConnect.

Capabilities:

- discover decks
- discover note models
- create notes
- update note fields
- add tags
- attach media
- query existing notes for dedupe or reconciliation

Important boundary:

- there is no assumption of direct AnkiWeb mutation
- browser-based editing of AnkiWeb should be treated as out of scope
- the intended path is local Anki plus normal Anki sync

Fallback workflows:

- CSV export
- media bundle export
- optional `.apkg` generation later

## Search Design

Search must be index-aware.

### Query Handling

- Hanzi input: query indexed glyph fields and exact/prefix matches in `character_summary`
- numbered pinyin input: normalize and query `k_mandarin`
- mixed input: query both glyph and reading paths and merge
- direct token, Unicode, or internal id lookup: bypass broad search
- component input: query decomposition views to find characters containing that component

### Ranking Signals

- exact character match
- exact reading match
- exact component match
- extracted frequency rank when available
- prefix over contains
- HSK level when available

### Search Service Responsibilities

- tokenize input
- normalize pinyin input
- choose indexed query path
- merge and rank results
- produce snippets and highlights
- support component containment queries against recursive decomposition views

## Parsing and Normalization

The backend must include a generated dictionary normalization layer.

Responsibilities:

- normalize glyph identity fields from `glyphs` and `character_summary`
- normalize readings, definitions, radical-stroke metadata, stroke count, HSK, and frequency
- parse `decomp_components` JSON and expose immediate decomposition data
- query recursive component views for base components and full decomposition trees
- emit display-safe HTML or structured JSON blocks for the frontend
- derive flashcard defaults from normalized entry data
- preserve source/build metadata from `metadata`

This parser should be versioned so cached render output can be invalidated when parsing logic changes.

## Cache Design

Caching should exist in three places.

### 1. Backend Query Cache

For repeated search and entry requests:

- key by normalized query and parser version
- store ranked result ids and summary payloads
- invalidate when parser version or source DB checksum changes

### 2. Normalized Entry Cache

For normalized character entries:

- key by glyph token or internal id and parser version
- store normalized JSON payload
- avoid reparsing decomposition JSON or rebuilding component trees on every request
- include decomposition analysis version in cache keys

### 3. Media Asset Cache

For downloaded and transformed assets:

- store original fetch metadata
- store derived variants for web and Anki
- deduplicate by checksum
- track expiry or refresh policy per source

### 4. Mnemonic Generation Cache

For personalized mnemonic generation:

- key by user id
- character
- component analysis version
- radical preference hash
- mnemonic generation profile id
- mnemonic structure template id
- model name and model version

This is important because the same character may need different mnemonic output for different learners.

Suggested cache policy:

- search results: short TTL
- normalized entries: persistent until parser or DB changes
- sourced media: persistent with periodic revalidation
- TTS or LLM fallbacks: persistent unless explicitly regenerated
- user-specific mnemonic generations: persistent until user preferences, prompt template, or model version changes

## Background Job Design

Use asynchronous jobs for any slow or failure-prone work:

- bulk media sourcing
- TTS generation
- LLM mnemonic generation
- media transcoding
- batch deck sync
- cache warmup

Each job should have:

- id
- type
- target scope
- status
- progress
- logs
- retry count
- last error
- rate-limit classification when throttled

The frontend should expose job status for transparency.

## Security and Compliance

- treat website-sourced media as content with license obligations
- store attribution and license status for each asset
- avoid using scraped material without a reviewable provenance record
- isolate provider credentials for TTS and LLM services in backend configuration
- do not expose provider secrets to the frontend
- treat mnemonic preferences and account data as private user data
